= Multi-tenancy with Orgs
:toc:
:toclevels: 1

:page-title: Multi-tenancy and orgs
:page-pageid: orgs
:page-description: You can now configure your ThoughtSpot instance as a mult-tenant cluster with separate Org containers for your tenants.

The multi-tenancy feature allows a ThoughtSpot cloud instance to be logically partitioned into multiple tenant-specific environments called `Orgs`. With Orgs, each tenant's data is isolated and protected with access control, and is invisible to the other tenants that share the same Thoughtspot application instance.

If the Orgs feature is enabled on your instance, your cluster admin can create an Org for each tenant account, configure groups and users, and control access to data objects. Each Org serves as an independent container with its own set of users and data, and provides the same user experience as that of a regular ThoughtSpot instance.

== Orgs in ThoughtSpot

If the Orgs feature is enabled on your instance, a *Primary Org* will be created by default. This default Org is referred to as `Org 0`. The administrator of the default Org can create one or several Orgs, and add an administrator user profile to each Org to manage users and groups, and define access control at the Org level.

The following diagram depicts the Orgs presentation on a multi-tenant ThoughtSpot instance.

image::./images/org-hierarchy.png[Multi-tenant ThoughtSpot environment]

[NOTE]
====
* The `Primary Org (Org 0)` cannot be deleted.
* Each Org is identified by a unique number and can have its own users and groups.
* Groups are unique to an Org and can be created only within the context of an Org.
* A user can belong to multiple Orgs.
====

== All Org scope

On a multi-tenant ThoughtSpot instance, all operations are run in the context of Org that the user is currently in. If a cluster administrator wants to perform a CRUD operation or apply a configuration change to all Orgs, they must use the `All Org` scope by setting the orgId to `-1`.

For example, to get a list of Orgs on a ThoughtSpot instance, the administrator must set the `orgId` parameter value to `-1` and pass it as a query parameter in the `GET /tspublic/v1/org/` API request. Similarly, to add a user to all Orgs or remove a user from all Orgs, you must set the orgID to `-1`.

== User roles

On a multi-tenant instance, ThoughtSpot supports the `Cluster administrator` persona. By default, the administrator of the primary Org (Org 0) is assigned the privileges of a cluster administrator.
The cluster administrator can access all Orgs and perform CRUD operations. The cluster administrator can also switch between Orgs and also act as an Org administrator.

Org administrators can administer and manage users and groups of their respective organizations. The users can belong to more than one Org and access data objects and metadata based on the privileges assigned to the groups to which they belong.

== Create and manage Orgs

To create an Org on a multi-tenant ThoughtSpot instance, you require cluster administrator privileges. By default, ThoughtSpot sets the administrator of the `Primary Org (Org 0)` as a cluster administrator. The cluster administrator can create Orgs using either the UI or the REST v1 API endpoints.

Creating and administering Org objects via REST API involves the following steps:

. Log in to your ThoughtSpot instance as a cluster administrator.
. To create an Org, send a `POST` request to the `POST /tspublic/v1/org/` with the Org name and description text in the request body.
. Set the Org context to `All Org` scope. The administrator must set the `orgId` to `-1` and pass it as a query parameter in the API request.
+
If the API operation is successful, ThoughtSpot creates an Org and assigns an Org ID.

. To view the ID assigned to an Org object, use the `POST /tspublic/v1/org/search` or `GET /tspublic/v1/org/` API endpoint.

To edit the properties of an Org object, you can use the `PUT /tspublic/v1/org/` endpoint. Note that the `PUT` request allows editing the name and description properties of an Org object and setting its status as active or inactive.

To delete an Org object, send a `DELETE` request to the `/tspublic/v1/org/` endpoint with Org ID or name in the request body.

For more information about REST V1 Org API endpoints, see xref:org-api.adoc[Org APIs].

== Switch between Orgs

A cluster administrator can access several Org objects and apply configuration changes. Before performing any operation on the objects in an Org, the administrator must set the Org scope and ensure that configuration changes are applied in the relevant Org context.

The cluster administrator can switch between Orgs by either using the Orgs drop-down in the UI or by sending a `PUT` request to the `/tspublic/v1/session/org/`.

For more information about switching between organizations using REST API, see xref:session-api.adoc#orgSwitch[Switch between Orgs].

== Manage users and groups

On a multi-tenant instance, the cluster and Org administrators can create and administer users and groups. The cluster administrator can perform CRUD operations on user and group objects of all Orgs, whereas the Org administrator can create and administer users and groups only in the Org context to which they belong.

[NOTE]
====
To perform CRUD operations on user and group objects of an Org, the cluster administrator must switch to the appropriate Org context.
====

=== Map a group to an Org

To associate a group to an Org object, you must specify the Org ID in the `POST` request to the `/tspublic/v1/group/` API endpoint.

For more information, see xref:group-api.adoc#create-group[Create a group object].

=== Add users to an Org

You can add a user to an Org when creating a user object using the `POST /tspublic/v1/user/` endpoint. If you want to add a bulk of users to an Org, use the `POST /tspublic/v1/user/addtoorg` API endpoint.

Note that you must specify the Org ID in your API requests.

For more information about user association to an Org object, see xref:user-api.adoc#addOrg[Add users to an Org].

=== Remove a user from an Org
To remove a user from an Org, you can use the `DELETE /tspublic/v1/user/{userid}` endpoint.

If a user belongs only to the Org specified in the API request, the user will be deleted from the Org and ThoughtSpot system. If the user belongs to more than one Org, the `DELETE` operation removes the user only from the Org specified in the API request. To remove the user from all Orgs and ThoughtSpot system, you must set the `orgid` to `-1` in your API request.

== User authentication

If your deployment uses the xref:trusted-authentication.adoc[trusted authentication] method to authenticate users, you will notice that the `POST /tspublic/v1/session/auth/token` endpoint allows you to send the `orgid` in the request body. If a user is mapped to a specific org, you can pass the `orgid` in the API request to obtain an authentication token and log in the user to the relevant Org context.

For more information about authentication token requests, see xref:session-api.adoc#session-authToken[Obtain an authentication token].
